# Добавление сверточной нейросети (CNN) в пайплайн маршрутизации банковских запросов  
! Чтобы избежать переобучения CNN, нужно больше обучающих данных (допустим, у нас есть 1000+ размеченных примеров)

## Гибридный подход
- **CNN** для быстрой обработки типовых запросов (уверенность ≥ порога)
- **LLM (YandexGPT)** для сложных/неоднозначных случаев (уверенность < порога)

## Архитектура CNN (1D)  

```
Вход: эмбеддинг размерности d
    ↓
Reshape: добавить канал → (1, d)
    ↓
Conv1D (фильтры=32, ядро=3, padding='same') + BatchNorm + ReLU
    ↓
MaxPool1D (окно=2)
    ↓
Conv1D (фильтры=64, ядро=3, padding='same') + BatchNorm + ReLU
    ↓
MaxPool1D (окно=2)
    ↓
AdaptiveMaxPool1D(1) → свернуть всю последовательность в один вектор
    ↓
Flatten → (64)
    ↓
Полносвязный слой (64 → 128) + Dropout(0.5) + ReLU
    ↓
Выходной слой (128 → число классов) + Softmax (вероятности)
```

## Пайплайн

```
1. Загрузить датасет с разметкой (8 классов)
2. Предобработка: нормализация, удаление пунктуации, лемматизация (опционально)
3. Векторизация (TF-IDF / более сложный векторизатор / предобученные эмбеддинги)
4. Разделение на train, validation и test (например, 0.7/0.15/0.15)
5. Создать DataLoader (batch_size=16-32)
6. Обучить CNN:
   - Оптимизатор: Adam (lr=0.001, weight_decay=1e-4)
   - Loss: CrossEntropyLoss
   - Эпохи: до 30, Early Stopping (patience=5)
   - Сохранять лучшие веса по валидационной accuracy
7. Гибридный порог (например, 0.7 - можно настроить адаптивно):
   - Если max(softmax) ≥ порога → ответ от CNN
   - Иначе → запрос в YandexGPT (few-shot промпт)
8. Оценка: precision, recall, F1, доля обращений к LLM

При применении в продакшене:
+ Кэширование ответов для повторяющихся запросов
+ Дообучение CNN на новых данных
```

## Вывод
Гибридный подход с CNN на данных достаточного объема (1000+ размеченных примеров) предположительно будет давать лучшую точность, чем LogReg (+ качество зависит от эмбеддингов), при все еще относительно низкой задержке (в сравнении с baseline - LLM only подходом) и контролируемых затратах на LLM. 

→ Предложенный пайплайн должен обеспечивать оптимальный баланс между скоростью, точностью и стоимостью и легко масштабироваться для применения в реальных банковских чат-ботах.
